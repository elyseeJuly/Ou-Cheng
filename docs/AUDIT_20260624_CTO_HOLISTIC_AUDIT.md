# 偶成 (Ou-Cheng) CTO 深度审计报告

> **审计日期**: 2026-06-24
> **审计人**: CTO (AI)

作为该项目的技术负责人，在对项目全局进行梳理和审计后，我整理了如下关于系统现状、健康度以及未来演进路线的深度报告。

---

## 1. 项目理解

- **项目定位**：一款专为现代创作者打造的“东方美学诗词创作与灵感探寻工具”。它不是简单的文本编辑器，而是结合了经典文化（平仄格律）与前沿数字技术（AI、3D）的沉浸式“数字书斋”。
- **用户价值**：
  - **专业性**：降低古诗词创作门槛，提供实时的平仄校验、韵脚预警。
  - **心流体验**：以极简、古典美学的 UI 设计（宣纸滤镜、金石印章）屏蔽现代化干扰。
  - **灵感赋能**：通过大模型（Gemini）点评及 3D 意象共现网络，打破创作瓶颈。
- **核心玩法/核心业务**：
  1. **挥毫（创作）**：核心的结构化格律输入与校验面板。
  2. **墨迹（作品）**：排版渲染、盖印章、高精无损导出卡片。
  3. **意象与卷帙**：本地古诗词库检索与 3D 动力学灵感发散词云。
- **技术栈分析**：
  - **表现层与框架**：React 19 + Vite 6 + TypeScript (无 Tailwind，使用原生 CSS Variables 与滤镜)。
  - **业务引擎**：自研正则表达式匹配与状态机，结合本地词典资源（平水韵、词林正韵等）。
  - **图形与交互**：Three.js（3D 意象贝塞尔曲线连线）。
  - **AI 赋能**：@google/genai (Gemini SDK) 纯前端调用。
  - **基础设施**：本地 LocalStorage 持久化，PWA 插件支持离线运行。

## 2. 系统全景图

- **模块划分**：
  - **UI/页面层**：`pages` 目录（Create, Works, Library, Settings）和 `components` 目录。
  - **领域引擎层**：`src/engine`（包含 `meterChecker.ts` 和 `rhymeData.ts`）。
  - **基础设施层**：`services`（提供大模型交互 `geminiService.ts`、存储逻辑 `storageService.ts`、导出 `exportService.ts`）。
- **数据流**：
  用户在 UI 组件输入 $\rightarrow$ 传递给领域引擎进行平仄和字数检测 $\rightarrow$ 检测结果实时反馈回页面渲染 $\rightarrow$ 用户手动触发保存 $\rightarrow$ 序列化后通过 `storageService` 写入浏览器 LocalStorage $\rightarrow$ 作品列表/导出模块读取并进行图像化（Canvas）。
- **控制流**：
  基于 React Router 驱动全局页面切换。核心创作页（Create.tsx）内部通过繁杂的 State（模式、体例、表单）进行条件渲染，根据不同的诗体挂载对应的格律模板引擎。
- **依赖关系**：
  视图层高度依赖本地的引擎校验逻辑；整体业务目前处于“单机重客户端”模式，强依赖浏览器端能力（DOM 转 Canvas 导出、LocalStorage 存储），外部依赖仅有 Gemini API 和前端的三方构建/UI库。

## 3. 项目健康度评分

基于现代工程标准，整体评分如下：

- **架构 (Architecture)：2/5（堪忧）**
  项目目录结构存在**非常严重的反模式**：`pages`, `components`, `services`, `App.tsx`, `index.tsx`, `types.ts` 全部堆砌在项目根目录，而不是在常规的 `src` 目录内。这破坏了前端工程化约定，未来容易导致打包配置灾难与维护混乱。
- **可维护性 (Maintainability)：2.5/5（较差）**
  核心逻辑组件 `Create.tsx` 是一个超过 600 行的“巨石组件（God Component）”，将状态管理、UI 渲染、AI 调用逻辑、数据保存强耦合在一起，充满了几十个 `useState`，牵一发而动全身。
- **可扩展性 (Extensibility)：3.5/5（尚可）**
  得益于 `src/engine` 与视图层有一定的分离，业务领域模型（词牌、格律模板）的扩展相对容易；但 UI 层面的状态没有解耦，扩展新的交互模式会很痛苦。
- **可测试性 (Testability)：3/5（及格）**
  值得表扬的是，底层的格律引擎（`meterChecker.ts`）配置了 Vitest 并有相应的单元测试；但除此之外的服务层、庞大的 UI 逻辑全无测试覆盖。
- **发布成熟度 (Release Maturity)：4/5（优秀）**
  作为一个无需后端的轻量级工具，它充分利用了 Vite, PWA, GitHub Pages 等现代化部署特性，开箱即用，运行速度快，发布链条非常成熟。

---

## 4. 最危险的 5 个问题

1. **根目录灾难与架构畸形**：所有的业务代码（组件、页面）都不在 `src` 中。这不仅让仓库看起来像个新手玩具，还可能在未来引入构建工具（如 Linter、Babel、Vite）路径解析时的潜藏 Bug。
2. **状态管理极度脆弱**：类似 `Create.tsx` 这种核心组件内部维护了极其复杂的派生状态。一旦用户提出“撤销/重做”或者“多端草稿箱同步”的需求，当前这种野生 `useState` 的架构将直接崩溃。
3. **前端直连大模型的安全隐患**：`geminiService.ts` 虽然设计了让用户自己填 API Key 的机制，但代码中存在 `import.meta.env.VITE_GEMINI_API_KEY` 的逻辑。如果开发者在本地不小心将自己拥有高额度的 Key 打包进生产环境，将会面临直接的资金盗刷风险。
4. **单点数据丢失危机**：目前的架构强依赖浏览器 `LocalStorage`。对于创作者而言，诗词文稿是核心资产。用户一旦清理缓存或更换浏览器，几个月的心血将瞬间灰飞烟灭，当前的 JSON 手动导出不足以兜底。
5. **DOM 转图的兼容性雷区**：采用 `html2canvas` 进行宣纸和印章的图像导出，这种方案在不同的移动端浏览器、不同的系统字体渲染下极易出现错位、白屏或截断，是未来客诉的重灾区。

## 6. 如果只能保留 30% 代码，保留什么？

- **保留内容**：
  1. `src/engine` 下的 **格律检验算法体系与韵书数据库**。
  2. 基于 Three.js 构建的 **3D 意象共现网络** 及相关的算法映射数据。
  3. UI 中的 **CSS 滤镜体系（宣纸纹理算法）和金石排版样式**。
- **删除内容**：所有的页面路由框架、组件状态逻辑（如 `Create.tsx`）、本地存储逻辑（`services`）。
- **原因**：
  作为一个“东方美学诗词工具”，它的**商业护城河**与**核心灵魂**在于**“领域知识（格律引擎）”**与**“独特的美学表现力（3D 意象与宣纸质感）”**。
  前端的 React 路由、状态管理、存储逻辑是业界标准化的“脚手架”代码，重写它们不仅成本低，而且随时可以用最新的框架（如 Next.js）做得更好。如果在绝境下只能留存精华，必定是保留这些不可替代的、打磨了极久的业务壁垒代码。

## 7. CTO 最终结论

从产品视角来看，**《偶成》是一件极具品味、克制且体验极佳的数字艺术品**。它精准切入了古诗词创作者的痛点，将生硬的格律平仄与优雅的心流体验完美融合。

但从工程视角来看，**项目目前是一座“盖在沙滩上的精美木屋”**。它的工程基建（目录规范、状态管理、数据持久化）远没有达到能支撑下一步商业化或规模化的标准。

**结论**：项目的技术路线（纯前端离线架构）非常契合其当前的“私密创作工具”定位，且未来扩展**没有根本性的死角**（无技术债死局）。但我们必须在下一个功能迭代前，按下暂停键，立刻执行“骨科手术式的重构”，夯实基础架构。只有这样，它才能真正从一个“优秀的 Demo”蜕变为一个“伟大的创作平台”。
