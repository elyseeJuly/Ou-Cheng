## 📖 一、 背景与设计痛点

在当前的「卷帙」（典籍库）模块中，所有的古典诗词数据均使用写死在 `constants.ts` 中的本地 Mock 数据（`MOCK_CLASSICS`）。为了提升应用底蕴，亟需将 GitHub 著名开源项目 `chinese-poetry/chinese-poetry` 里的海量唐诗和宋词数据平滑集成至应用中，同时保证极致的页面渲染速度和离线使用体验。

### 🚨 核心痛点
1. **数据体量与加载速度瓶颈**：唐诗宋词原始 JSON 动辄几十兆，若一次性静态打包入客户端会导致打包体积剧增、首屏加载白屏。
2. **网络依赖问题**：本应用主打“沉浸式纯本地心流创作”，必须完全离线可用，不能在阅读时发起动态外网网络请求，以免产生卡顿。
3. **数据结构不统一**：开源库中的唐诗和宋词格式字段并不一致（如宋词采用 `rhythmic` 表示词牌，唐诗采用 `title` 表示标题），需要统一清洗。

## 🛠️ 二、 技术实施方案

### 1. 数据获取与清洗管道 (Crawler & Cleaner)
采用 Node.js 编写一个独立的数据拉取清洗脚本 [fetch-poetry.js](file:///Users/quantumrose/Documents/Emberois/ou-cheng/scripts/fetch-poetry.js)。
- 从 `chinese-poetry` 项目拉取全唐诗 (`poet.tang.0.json`) 和宋词 (`ci.song.0.json`) 的首批核心数据（各限制 500 首以控制包体大小）。
- 统一清洗为 `{ title, author, content, category }` 的结构。
- 最终输出到 `public/data/poetry.json` 中作为静态资源托管，实现本地离线加载。

### 2. 离线懒加载与状态合并 (UI Lazy-Load & Merge)
修改「卷帙」模块的主体页面 [Library.tsx](file:///Users/quantumrose/Documents/Emberois/ou-cheng/pages/Library.tsx)。
- 在组件挂载（`useEffect`）时，采用后台懒加载（`fetch`）方式动态异步获取 `public/data/poetry.json`。
- 在加载完成前展示精致且充满禅意的 Loading 骨架屏。
- 加载完成后，将获取到的数据与本地基础 `MOCK_CLASSICS` 进行无缝合并，注入全局搜索上下文，确保用户搜索即时响应。

## 🚫 四、 AI 执行约束与红线

> [!IMPORTANT]
> 1. **强制类型安全**：抓取并清洗后的数据必须 100% 兼容 [types.ts](file:///Users/quantumrose/Documents/Emberois/ou-cheng/types.ts) 中定义的 `ClassicPoem` 接口契约。
> 2. **极致用户体验**：获取诗歌 JSON 必须完全异步，禁止在主线程进行 CPU 密集型同步解析导致 UI 帧率骤降。
> 3. **极简依赖**：爬取和清洗逻辑必须使用 Node.js 原生 `https` 模块，禁止引入第三方体积庞大的 HTTP 库（如 axios），以保持项目轻量化。
